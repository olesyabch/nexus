# Технологический стек проекта Nexus

## Общая концепция

Для реализации проекта Nexus использован современный конвейер разработки, охватывающий все этапы: от проектирования и управления до автоматической генерации кода. Стек представляет собой сочетание low-code инструментов и технологий искусственного интеллекта.

---

## 1. Платформа управления проектами и задачами (WEEEK)

**Назначение:** Организация командной работы и контроль выполнения задач.

**Применение в проекте:** WEEEK использовался как единый центров планирования. В системе велись спринты по методологии Agile: создавались задачи для каждого этапа (дизайн, база данных, документация), назначались ответственные (Милена, Алиса, Олеся, Настя), устанавливались дедлайны и отслеживался прогресс по Kanban-доске.

**Результат:** Обеспечено прозрачное управление проектом и координация распределённой команды.

---

## 2. Проектирование архитектуры (Miro)

**Назначение:** Визуальное моделирование архитектуры информационной системы и бизнес-процессов.

**Применение в проекте:** Miro использовался для создания полной архитектурной схемы проекта, включающей:

- **Слой бизнес-логики:** Разработаны схемы API Gateway (авторизация, маршрутизация запросов), Matching Service (алгоритмы поиска и сортировки менторов), Transaction Manager (механизмы списания, заморозки и начисления внутренней валюты Skill Coins), Message Broker (обработка P2P-сообщений и чатов между пользователями).

- **Слой данных:** Спроектирована структура баз данных: Users DB (личные данные, рейтинг и отзывы), Transactions DB (баланс Skill Coins, история изменений, логи покупок), Chats DB (история переписки).

- **Слой клиентского приложения:** Визуализированы ключевые экраны: личный кабинет, лента поиска с фильтрами, кошелёк с историей транзакций, раздел чатов.

Ключевая особенность Miro — возможность совместной работы в реальном времени, где вся команда одновременно редактировала схемы, оставляла комментарии и вносила правки.

**Результат:** Визуально задокументированная трёхуровневая архитектура (клиентское приложение — бизнес-логика — данные), которая стала основой для проектирования базы данных в Supabase, написания технического задания и распределения задач между разработчиками.

## 3. Дизайн и прототипирование (Figma)

**Назначение:** Создание дизайна интерфейсов и интерактивных прототипов.

**Применение в проекте:** Figma использовался для создания полноценного кликабельного прототипа веб-сайта (Low-Fi и Hi-Fi). Проработаны все экраны: от регистрации до личных кабинетов. Ключевая особенность — возможность одновременной работы команды, где дизайнеры видели правки друг друга в реальном времени.

**Результат:** Готовый протестированный макет, служивший точным техническим заданием для разработки, что исключило необходимость дополнительного согласования между дизайнерами и разработчиками.

## 4. База данных и Backend-инфраструктура (Supabase)

**Назначение:** Предоставление готовой backend-инфраструктуры (Backend-as-a-Service) на основе PostgreSQL, объединяющей базу данных, аутентификацию и API.

**Применение в проекте:** Supabase является центральным хранилищем всех данных о пользователях, навыках, уроке и транзакциях.

**Ключевые аспекты использования:**

- **Реляционная база данных:** Спроектирована нормализованная структура таблиц (`profiles`, `universities`, `institutes`, `group`, `lessons`, `skills_list`), связанных внешними ключами для обеспечения целостности данных.

- **Безопасность:** Настроены политики Row Level Security (RLS), гарантирующие, что любой пользователь может просматривать профили других (для поиска менторов), но изменять может только свой собственный профиль. В таблице уроков обычный пользователь видит и изменяет только свои уроки (где он учитель или ученик), а администратор имеет полный доступ ко всем данным.
- **Автоматизация:** Использованы триггеры базы данных (например, для автоматического создания профиля при регистрации) и встроенная аутентификация (регистрация и вход по email и паролю).

**Результат:** Получен надёжный масштабируемый бэкенд без необходимости настройки и поддержки собственного сервера, что позволило сфокусироваться на бизнес-логике приложения.

## 5. Код и документация (GitHub)

**Назначение:** Хостинг кода, версионный контроль и совместная разработка.

**Применение в проекте:** GitHub выполняет функцию единого источника для всей цифровой части проекта:

- **Версионный контроль кода:** Хранилище для HTML/CSS/JS кода, сгенерированного через ИИ.
- **Документация:** Техническая документация проекта ведётся в формате Markdown непосредственно в репозитории, что обеспечивает её актуальность и доступность для всех участников команды.

**Результат:** Обеспечена прозрачная история разработки, позволяющая любому члену команды отслеживать изменения проекта на любом этапе.

---

## 6. Генерация кода (Qwen)

**Назначение:** Большая языковая модель Qwen, способная генерировать код по текстовому описанию и изображениям.

**Применение в проекте:** Созданный в Figma Hi-Fi прототип (полноцветный, детализированный макет) загружался в Gwen. Вместе с визуальным материалом в модель отправлялись детальные цепочки сценариев использования: подробное описание логики переходов между экранами, действий пользователя и требований к функционалу. На основе этого Qwen генерировал готовый HTML/CSS/JS код, который затем помещался в репозиторий GitHub.

**Результат:** Применён передовой подход «AI-трансформация дизайн-кода», где ИИ выступает полноценным инструментом разработки, значительно ускоряющим создание MVP за счёт автоматической генерации кода по визуальным макетам и сценарным описаниям.
