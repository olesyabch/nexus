# Техническое задание (ТЗ) на разработку платформы взаимного обучения «Nexus»

## 1. Общие сведения

**Наименование проекта:** Веб-платформа взаимного обучения «Nexus»

**Назначение системы:** Организация внутривузовской экосистемы обмена знаниями (peer-to-peer обучение) на основе бартерной экономики.

**Целевая аудитория:** Студенты и преподаватели высших учебных заведений (основные участники), администрация институтов (контролирующая роль).

---

## 2. Архитектура и технологический стек системы

**Архитектура:** Трёхзвенная (Фронтенд — Слой бизнес-логики — База данных/Бэкенд)

**СУБД и Бэкенд-инфраструктура:** BaaS Supabase на базе реляционной СУБД PostgreSQL

**Фронтенд-компоненты:** Семантическая верстка HTML5, стили CSS3, динамические скрипты управления интерфейсом на JavaScript, сгенерированные при помощи БЛМ Qwen по Hi-Fi макетам из Figma

**Система версионного контроля:** GitHub

---

## 3. Функциональные требования к системе

Система должна быть разделена на два ключевых контура автоматизации в зависимости от роли пользователя.

### Контур 1: Личный кабинет Участника (Студент / Преподаватель)

**Аутентификация:** Регистрация и вход по связке Email и Пароль через модуль Supabase Auth с автоматическим триггером создания записи в таблице profiles.

**Многоуровневый онбординг:**
- Фильтрация пользователей на «Частные» и «Связанные с университетом»
- Привязка участников к академической структуре (Университет, Институт)
- Индивидуальное ветвление: обязательный выбор учебной группы для Студента; автоматический пропуск этого шага для Преподавателя

**Управление навыками (Мэтчинг):** Интерфейс выбора компетенций из таблицы skills_list, разделенный на два блока: «Чему могу научить» (роль Ментора) и «Чему хочу научиться» (роль Ученика).

**Внутренняя экономика (Skill Coins):** Реализация логики холдирования (заморозки) внутренней валюты на время проведения урока и автоматического списания/начисления коинов после подтверждения успешности занятия.

**Коммуникация:** Наличие модуля P2P-чатов для обсуждения деталей занятий.

### Контур 2: Панель Администратора университета

**Авторизация:** Вход в закрытый дашборд по специальному коду/токену верификации прав.

**Безопасность данных (RLS):** Администратор имеет сквозной доступ к чтению таблиц уроков (lessons) и профилей (profiles) для контроля активности, но не имеет доступа к изменению личных данных пользователей.

**Аналитический модуль:** Мониторинг проведённых занятий, выгрузка отчетов по востребованности конкретных навыков внутри институтов и учебных групп для деканата.

---

## 4. Требования к надежности и безопасности

**Защита данных на уровне строк:** Использование политик Row Level Security (RLS) в Supabase. Пользователь может редактировать только свою запись в таблице profiles.

**Целостность данных:** Обеспечение связей между таблицами (profiles, universities, institutes, groups, lessons, skills_list) с помощью внешних ключей (Foreign Keys).

**Масштабируемость:** Архитектура СУБД PostgreSQL должна позволять бесшовный переход с бесплатного тарифа Free-tier на коммерческий при увеличении нагрузки.

---

## 5. Состав и стадии разработки MVP

**Стадия 1:** Сбор требований и проектирование архитектуры в Miro, ведение спринтов в WEEEK.

**Стадия 2:** Разработка дизайн-системы, UI-kit и интерактивного Hi-Fi прототипа в Figma.

**Стадия 3:** Проектирование реляционных таблиц и логики безопасности в Supabase.

**Стадия 4:** Автоматическая генерация фронтенд-кода через нейросеть Qwen и связка с API бэкенда.

**Стадия 5:** Развертывание кодовой базы на GitHub, тестирование и подготовка к пилотному запуску.
