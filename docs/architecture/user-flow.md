# Архитектура пользовательских сценариев (User Flow)

Логика регистрации, авторизации и распределения ролей в платформе "НЕКСУС" реализована на стороне клиента с помощью кастомных скриптов динамического ветвления (JavaScript), сгенерированных моделью Qwen, и интегрирована с облачной базой данных Supabase (PostgreSQL). Система имеет трёхуровневую систему фильтрации и ветвления интерфейса на этапе онбординга.

---

## 1. Точка входа и первичная авторизация (Модуль Supabase Auth)

**Вход в систему (Авторизация):** Пользователь вводит связку «Фамилия, имя, никнейм, email и пароль». Запрос отправляется в модуль аутентификации Supabase. При успешной валидации данных JavaScript перенаправляет сессию на Главную страницу (Дашборд).

**Инициация регистрации:** При создании нового аккаунта пользователь проходит первые шаги.

**Шаг 1 (Создание аккаунта):** Ввод Фамилии, имени, никнейма, email и пароля. Данные уходят в Supabase Auth. На стороне БД срабатывает автоматический триггер, который создаёт пустую запись в таблице profiles.

**Шаг 2 (Персонализация профиля):** Пользователь заполняет блок «О себе» (описание компетенций и интересов). Данные записываются в таблицу profiles.

---

## 2. Три уровня логического ветвления (if/else в коде)

После заполнения базового профиля JavaScript запускает последовательную фильтрацию пользователя.

### УРОВЕНЬ 1: Разделение на «Частные» и «Связанные с университетом» пользователи

Система задаёт первый вопрос: «Вы связаны с университетом?»

**Вариант «НЕТ» (Частный пользователь):** JavaScript полностью блокирует академический блок. В таблицу profiles в поле user_type записывается значение private. Пользователь сразу перенаправляется на финальный экран выбора компетенций («Выбор навыков»).

**Вариант «ДА» (Академический сектор):** Система пропускает пользователя на следующий уровень проверки.

### УРОВЕНЬ 2: Разделение внутри ВУЗа (Преподаватель/Студент или Администратор)

Для пользователей, связавших себя с ВУЗом, открывается выбор: «Ваша роль в университете?»

**Вариант «Администратор»:** Переход на экран ввода проверочного токена/кода верификации прав модератора и привязки к конкретному Университету. Пропуск всех остальных шагов. В таблицу profiles записывается роль admin, открывается доступ к закрытому дашборду аналитики благодаря политикам Row Level Security (RLS).

**Вариант «Преподаватель / Студент»:** Система понимает, что это участники обмена знаниями. Они переходят на совместный экран, где вместе выбирают свой Университет и конкретный Институт/Факультет (данные подтягиваются из таблиц universities и institutes).

### УРОВЕНЬ 3: Разделение участников («Отлепление» Преподавателя от Студента)

После выбора института JavaScript считывает подтип пользователя, и пути разделяются.

**Сценарий «Студент»:** Направляется на обязательный шаг «Поиск группы». Студент выбирает свою академическую группу из таблицы groups для точного мэтчинга внутри потока.

**Сценарий «Преподаватель»:** Экран выбора группы автоматически скрывается на уровне кода. Преподаватель пропускает этот шаг.

---

## 3. Финал онбординга

После прохождения третьего уровня и Студент, и Преподаватель (а также Частный пользователь, пришедший с Уровня 1) снова соединяются на едином экране выбора навыков.

Шаг разделён на два блока:
- **«Чему могу научить»** — формирует профиль ментора
- **«Чему хочу научиться»** — запросы на обучение

Данные фиксируются в таблице profiles в колонках skills_can_teach и skills_want_learn.

По окончании в таблицу profiles участникам ВУЗа записывается роль user с полным доступом к маркетплейсу, P2P-чатам и кошельку Skill Coins.
